## 二、踩坑路线图（做错的和做对的）

### ❌ 踩坑 1：双向同步，服务器为权威源

**做法**：`loadServerData()` 中，合并时删除本地有但服务器没有的项。

**结果**：用户刚注册、服务器还没数据时 → 本地全部错题被清空。

**教训**：不能假设服务器数据完整。首次登录/同步未完成时，服务器空数据不应被视为"应该删掉"。

### ❌ 踩坑 2：双向同步 + 守卫条件

**做法**：只在服务器有数据时才做双向同步。

**结果**：解决了空数据清空问题，但删除操作是异步的 → 用户刷新页面时 DELETE 还没到服务器 → loadServerData 从服务器拉回被删的数据。

**教训**：异步删除的竞态条件，仅在服务器端做延迟处理不够。

### ❌ 踩坑 3：删除黑名单

**做法**：在 localStorage 维护 `_deletedSyncKeys` 黑名单，删除时标记，合并时跳过。

**结果**：理论上正确，但 `/user` 页面（user.html）有自己的删除函数，完全没有走黑名单机制。`deleteFavoriteItem` 直接操作 localStorage、`deleteHistoryItem` 用索引定位（删除后索引漂移）、`clearAll*` 清空时也不调用服务器端清理。

**教训**：同一个功能在两个页面有各自的实现，容易漏掉同步更新。应该把删除逻辑统一到 utils.js 中。

### ❌ 踩坑 4：版本号没更新

**做法**：改了 JS 代码但没更新 `?v=` 版本号。

**结果**：浏览器一直跑旧版 JS，用户反馈"没有实现"。中间反复重启服务器都没用。

**教训**：改 JS 文件后必须同步更新所有模板中的版本号参数，否则缓存会掩盖一切改动。

### ✅ 最终方案：单向同步（本地 → 服务器）

**做法**：`loadServerData()` 完全不做数据合并。本地永远是权威源，服务器只做备份。

**结果**：删除永不回弹。简单、可预测、容易推理。

---

## 三、做得好 vs 做得不好

### 👍 做得好

- **CSRF 防护**：早期就加上了 `X-Requested-By: QuizApp` 头部校验，避免了后续加端点时的安全遗漏
- **按需加载**：html2canvas 改为点击按钮时才加载，不阻塞页面初始化
- **版本号统一升级**：发现问题后对所有模板做了统一处理，一次性解决了缓存问题
- **代码审查**：用 review skill 检查发现了嵌套 `<script>` bug 等自己没注意到的问题
- **数据模型设计**：三个表的唯一约束、to_dict 方法、字段类型都合理

### 👎 做得不好

- **过度设计**：一开始追求完美的双向同步，花了太多时间在黑名单、守卫条件等复杂机制上。最终方案反而是最简单的单向同步
- **没有一次性覆盖所有页面**：`/user` 页面有独立的 JS 实现，改 `app.js` 和 `utils.js` 时完全没意识到还需要改 `user.html`
- **缓存意识不足**：改了 JS 没第一时间更新版本号，导致多次"修复无效"的假象
- **没有加版本号自动化**：每次手动改版本号容易遗漏，应该用自动化工具或统一版本变量
- **删除历史用索引**：user.html 的 `deleteHistoryItem(idx)` 用数组索引定位记录，删除后索引偏移会导致删错行。应该用记录的唯一 `id`

---

## 四、待改进

1. **跨设备数据同步**：当前单向同步（本地→服务器），换设备后需要手动拉取。后续可加"从云端恢复"按钮
2. **唯一操作入口**：删除、收藏等操作应统一到 utils.js，各页面只负责 UI 渲染，不直接操作数据
3. **版本号自动化**：Flask 的 `url_for('static', filename='xxx.js', v=VERSION)` 中的 VERSION 应该从环境变量或文件 hash 自动生成
4. **删除历史用 ID 而非索引**：user.html 的 deleteHistoryItem 还有索引漂移的隐患
5. **测试**：9 个 Sync API 端点没有自动化测试

## 三、经验教训

- **邮箱是最难搞的部分**：域名注册、DNS 配置、API 兼容性、反垃圾策略——每一层都可能卡住
- **curl 正常 ≠ Python 正常**：同样 URL、同样 Header、同样 Body，`urllib` 和 `curl` 行为不同。遇到这种问题果断换库
- **域名信誉度需要时间**：新注册的域名发 QQ 邮箱大概率被拒，这是正常现象，等几天就好
- **API Key 要重新生成**：在 Resend 加了新域名后，旧 API Key 不一定自动获得新域名的权限

---

## 一、本次改动

### 1. 题库持久化（核心）

**问题**：题库存在 `data/*.json` 文件，Railway 每次 Redeploy 重建文件系统，所有上传的题库全部丢失。

**解决方案**：新增 `QuestionBank` 数据库表，所有题库直接存到 PostgreSQL。

| 改前 | 改后 |
|------|------|
| `data/{bank_id}.json` 文件读写 | `question_banks` 表，`data_json` 字段存完整 JSON |
| `load_bank_list()` 遍历磁盘文件 | `QuestionBank.query.all()` 查数据库 |
| `load_bank()` 读 JSON 文件 | `QuestionBank.query.get(bank_id)` 查数据库 |
| `save_bank()` 写 JSON 文件 | `db.session.add(qb)` 写数据库 |
| 部署后数据丢失 | **数据持久化** ✅ |

**迁移逻辑**：启动时自动扫描 `data/*.json` 导入到数据库（已导入的跳过）。

### 2. 统一版本号

新增 `version.py`，所有模板通过 `{{ VERSION }}` / `{{ VERSION_NAME }}` 引用，以后改版本只改一个文件。

```python
# version.py — 唯一的版本号源
VERSION = "1.1.0"
VERSION_NAME = "v1.1.0"
```

### 7. 跨设备数据恢复

- 新增 `restoreFromServer()` 函数，调用 `GET /api/sync/all` 拉取云端数据合并到 localStorage
- 用户中心侧边栏新增「☁️ 从云端恢复」按钮
- 合并策略：错题本/收藏按 `question_key` 覆盖，学习记录按 `id` 去重追加
- 确认弹窗展示云端数据量（N 道错题、M 个收藏、K 条记录），确认后执行

### 8. 版本号自动化

- 将 10 个模板中 22 处硬编码 `v=4` / `v=3` 全部替换为 `v=VERSION`（`VERSION` 由 `version.py` 注入）
- 修复 `reading.html` CSS 版本号漏升 Bug（原为 `v=3`，其余均为 `v=4`）
- 以后改版本只需修改 `version.py` 一行

### 9. 删除历史索引漂移 Bug 修复

- `deleteHistoryItem` 改用 `record.id` 替代数组下标定位
- 修复连续删除多条记录时的索引错位 Bug
- 渲染时新增兜底 `hid = h.id || \`idx-${i}\`` 兼容无 ID 的老记录

### 10. 统一操作入口

- 将 user.html 中 5 个重复的内联数据操作函数收归 `utils.js`
  - `removeFavoriteItem(key)` — 删除单条收藏
  - `removeHistoryItem(id)` — 删除单条学习记录
  - `clearAllFavoritesLocal()` / `clearAllWrongBookLocal()` / `clearAllHistoryLocal()` — 清空三类数据
- user.html 只保留 confirm + DOM 动画，数据操作委托给 utils.js
- `deleteWrongItem` 之前已调用 `removeFromWrongBook(key)`，保持一致

### 11. 自动化测试

- 搭建 pytest 测试框架，使用 SQLite 内存数据库隔离运行
- 测试文件结构：
  - `tests/conftest.py` — 测试基础设施（app/client/user fixtures）
  - `tests/test_sync_api.py` — 26 个测试，覆盖 9 个 Sync API 端点
  - `tests/test_permissions.py` — 18 个测试，覆盖三级权限边界
- 测试内容包括：
  - CSRF 防护校验（缺头/错头 → 403）
  - 错题本/收藏/学习记录的增删查改 + 去重策略验证
  - 未登录访问 10 种组合 → 302 重定向
  - 访客/普通用户/管理员三级权限边界
- 运行方式：`python3 -m pytest tests -v`
