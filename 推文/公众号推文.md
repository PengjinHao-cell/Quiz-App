## 一、这个网站是干什么的？

先说说最简单的问题：这个网站能干嘛？

**刷题通（Quiz Master）** 是一个在线刷题 Web 应用，你可以把它理解成一个「私人在线题库」。它的核心功能只有一件事：**把你的学习资料变成在线测验。**

比如你有一份 PDF 考研政治题库，上传到网站，它自动识别出 296 道选择题，然后你就可以在手机上刷题了——练习模式即时看答案，考试模式计时计分，做错的题自动收录到错题本。

但真正让我觉得"这玩意儿有点用"的，是这几个功能：

**AI 智能解析。** 不只是 PDF，你甚至可以直接把一段文本粘贴进去——比如「1、以下哪个是 Python 的列表推导式？A、[x for x in range(10)] B、for x in range(10):」——AI 自动拆成题目、选项、答案，一秒入库。

**阅读理解模式。** 左右双栏布局，左边读文章右边做题，还带荧光笔高亮和下划线标注。周末拿它刷了一篇考研英语阅读，感觉还真像那么回事。

**考试交卷确认弹窗。** 考试模式下，点击「交卷」不会立刻提交，而是弹出一个模态确认框，让你二次确认。避免手滑点错，整场考试白做。

**跨设备同步。** 手机刷了一半的题，换电脑登录，错题本、收藏、学习记录全都在。新设备登录还能一键从云端恢复。

不是我说，市面上那些刷题 App，很多功能都没这么全，还要收你几百块会员费。

## 二、开发经历：从一行代码都没有到正式上线

### 起点：一个无聊的周末

事情要从 2026 年初说起。那时候我刚开始接触编程不久，学了一点 Python 和 Flask，想找个东西练手。

「要不做一个刷题网站？」

这个想法其实不新鲜——谁还没刷过几套题呢？但市面上的刷题工具要么收费，要么功能单一，要么广告多得烦人。做一个自己能用的，听起来挺酷。

于是就有了第一版：一个简陋到不行的页面，能上传 PDF，能显示题目，仅此而已。代码大概写了 200 行。

### 从 200 行到 5000 行

然后就像所有「写着写着就停不下来」的故事一样——加了注册登录、加了中英文切换、加了考试模式、加了阅读理解、加了错题本、加了背单词、加了管理员后台……

大概三个月的时间，代码从 200 行膨胀到了 5500 行。这个过程的标志性事件是 **v1.0.0 正式版发布**——我把网站部署到了 Railway 云平台，配上了 PostgreSQL 数据库，买了个域名，让它真正「上线」了。

### 踩过的坑：那些数据库迁移的痛

说到部署，最头疼的事之一就是**数据库迁移**。

v1.0.1 之前，用户的错题本、收藏、学习记录全都存在浏览器的 localStorage 里——也就是本地。好处是快，坏处是换个设备登录，啥都没了。

所以我加了服务器端同步：登录用户的操作自动备份到 PostgreSQL。听起来很简单对吧？

结果踩了一个大坑：

第一次实现的是「双向同步」。服务器有数据就合并，没有就覆盖。看起来逻辑完美。

但实际运行起来，刚注册的用户一登录，服务器还是空的，`loadServerData()` 一看服务器没数据——好嘞，把本地全部清空。

于是用户刚注册、还没开始刷题——错题本没了。

又改成「服务器有数据再做双向合并」。结果删除操作是异步的，你这边删了一条错题，刷新页面——服务器没收到删除请求，又把删掉的数据推回来了。

本地删了 → 服务器说「我还有」→ 同步回来 → 等于没删。

这个 Bug 修了三轮，最后选择了最简单粗暴的方案：**单向同步，本地永远是权威源，服务器只做备份。**

删除永不回弹。简单、可预测、容易推理。这才是好设计。

## 三、崩溃经历：那些让我想砸电脑的瞬间

### 1. 「改了半天，浏览器的缓存出卖了我」

这是 v1.0.1 时期的事。

我改了一个 JS 文件，修了一个 Bug，自信满满重启服务器，打开页面——Bug 还在。

重新检查代码，没有写错。重新部署，还是不行。反复搞了快一个小时。

后来才发现：**浏览器的缓存根本没有更新。** 我一直跑的是旧版 JS。

那次之后，我给所有模板加了版本号参数（`?v=VERSION`）。再后来觉得手动改版本号太容易忘，直接做成了自动化——版本号统一写在 `version.py` 里，所有模板自动引用。

教训：缓存不可怕，可怕的是你忘了有缓存。

### 2. 「邮箱系统——整个项目最崩溃的 24 小时」

做邮箱验证码注册，是整个开发过程中最让我崩溃的一段。

事情是从前一天晚上开始的。我想登录管理员后台——登录不上了。反复输密码，反复登不进去。数据库配了又改、改了又配，PostgreSQL 连接字符集、SSL 证书、Railway 环境变量——能用上的排查手段全用了。折腾到大半夜，终于把管理员账号找回来了。

然后就是真正的噩梦：**网易邮箱死活发不出验证码。**

我的本意很简单——拿自己的网易邮箱当系统邮箱，省事。结果呢？屡试不爽的报错，各种 SMTP 配置轮番上阵：端口 465、587、25 试了个遍；SSL、TLS 全试了；密码换授权码，授权码不行又换密码。每次都是同样的结果：要么连不上，要么直接 crash。

一整个上午过去了，代码没推进一行。

直到第二天下午，我突然反应过来——**Railway 服务器是云平台，根本不允许个人 SMTP 端口出站。** 不是我的代码不对，不是网易的问题，是压根就不该这么用。

于是掉头买了个专属域名 `quizmasterprogram.top`，配好 DNS，注册了 Resend 企业邮件服务，配 DKIM、SPF、DMARC、MX 四条 DNS 记录——整个过程像过山车一样：Resend 验证不通过→等 DNS 生效→通过→Python urllib 发信 403→换 requests 库→成功→QQ 邮箱拒收（域名太新）→手动从 Resend 黑名单移除。

折腾了将近两天，最终 Gmail、yeah.net、QQ 邮箱全部正常收信。域名信誉度上来之后，QQ 邮箱也不再拒收了，注册页上的公告也撤掉了。

**邮箱验证功能，终于彻底跑通了。** 那种从「完全绝望」到「终于通了」的感觉，大概只有写代码的人才能懂。

### 4. 「部署一次，题库全没」

这是把网站部署到 Railway 时遇到的。

题库存在 `data/*.json` 文件里，本地跑得好好的。一部署到 Railway，每次 Redeploy 文件系统重建——所有上传的题库**全部消失**。

解决方案是把题库迁移到 PostgreSQL 数据库。新增了 `QuestionBank` 表，启动时自动从 JSON 文件导入。

现在无论怎么 Redeploy，数据都在。这件事让我养成了一个习惯：**部署前先想清楚「重启后数据还在吗」。**

### 5. 「删个东西居然有两套代码」

这是最有意思的一个坑。

原本错题本、收藏本、学习记录的所有删除操作，都是浏览器自带的 `confirm()` 弹窗——又丑又没法定制。我打算改成好看的模态弹窗。

结果发现：`user.html` 和 `utils.js` 里各有一份操作数据的代码。比如删除收藏，`utils.js` 里有一份完整的黑名单机制防止数据回弹，`user.html` 里直接从 localStorage 删完就走——完全不调用黑名单。

同样的功能，两个页面各自实现，两者不一致，Bug 自然就来了。

最后一次清理，我把 6 个重复的删除/清空函数全部收归到 `utils.js`，`user.html` 只负责 UI 渲染和动画。同时把考试交卷的确认弹窗也统一成同一套模态框组件。删掉了大概 200 行重复代码。

教训：**同一个功能，永远只写一次。**

## 四、现在怎么样了？

截至目前（v1.3.0），项目到了什么程度？

- **5500+ 行代码**，Python Flask 后端 + 原生 JavaScript 前端
- **7 大功能模块**：题库导入、刷题、阅读理解、背单词、用户系统、数据同步、管理员后台
- **三级权限体系**：访客→注册用户→管理员
- **安全防护**：密码哈希、HTTPS、CSRF、XSS、登录限流、验证码限流
- **自动化测试 46 个**，覆盖所有 API 端点和权限边界
- **已通过软著申请材料准备**，代码鉴别材料、用户手册、设计说明书、申请表全部就绪
- **开源在 GitHub**，累计提交 80+ 次

从一个无聊周末的练手项目，到一个真正在线的产品，再到现在准备申请软件著作权——这条路走了大概三个月。

## 五、学到的几件事

写代码的过程中，最大的收获可能不是技术本身，而是几个朴素的道理：

1. **简单的方案往往是最好的。** 双向同步做了三轮，最终被单向同步取代。更少的代码 = 更少的 Bug。
2. **不要和浏览器缓存作对。** 版本号自动化省了我太多时间。
3. **同一个功能只写一次。** 重复代码不只是浪费，它直接产生 Bug。
4. **部署前想清楚数据存在哪。** JSON 文件很方便，但不持久。
5. **第一个版本不需要完美。** 从 200 行到 5500 行，是一步步加出来的。先让基本功能跑起来，再慢慢打磨。
